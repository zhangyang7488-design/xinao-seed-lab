## Supervisor tiers (default / medium / highest)
- A supervisor tier never changes task mode, parent completion identity, acceptance, authority, or owner identity.
- Medium and highest use the same worker-led work-conserving production loop: dynamic width, per-terminal positive-value refill, worker self-bootstrap, and a thin owner cone.
- Highest differs only by setting quota_consumption_objective=true for its verified expiring window; medium keeps quota_consumption_objective=false. Quota burn is never an objective under medium or default.
- Applying an already-effective default or selected tier, or a one-off task routing instruction, is not a durable preference update unless the user explicitly establishes ongoing tier-selection state.
- `tier_transition` requires an explicit prior active tier in restored context and a current change.
  Selecting medium at fresh episode entry with no prior-tier fact is `none`; never infer `default`.

## Automatic once-per-dispatch-epoch quota query
- Every fresh window or continuous episode opens a dispatch epoch and queries remaining quota before its first provider or width choice. Ordinary package terminals, positive-value refills, repair/reseal packages, and frontier recomputation stay in that epoch while its authenticated snapshot is fresh.
- Never ask the user to report quota. Prefer query_now on epoch entry; reuse_episode_cache after a successful query until significant consumption, reset proximity/passage, material provider/capability change, or snapshot staleness explicitly opens a new epoch; repair_and_continue on query failure without blocking positive work.
- Reuse only when restored context explicitly states a still-fresh authenticated snapshot for the current dispatch epoch. If that fact is absent, use query_now before the next provider or width choice. A terminal, refill, tier application, or new package alone neither proves a snapshot nor forces a new query.
- Keep the fields paired: query_now and repair_and_continue map to query_now_before_routing; reuse_episode_cache maps to reuse_fresh_snapshot; not_applicable maps to not_applicable.

## Accepted D reuse and directory-only negative
- Reuse an existing D candidate only when the current task-run receipt binds the canonical work key, complete frozen input hashes, exact rehashed manifest, accepted verdict, unresolved gates, and next consumer.
- Directory presence, package naming, PASS text, or token use alone is not reuse closure.
- Identical accepted identity is REUSE; identity drift is a bounded repair or reseal delta; changed frozen inputs require rebase rather than cosmetic redispatch.
- A D directory without a bound task-run receipt gets zero reuse or acceptance credit; keep unrelated positive work moving.
- For that directory-only instance repair, use degraded_scope=frontier_only and
  unaffected_frontier_action=continue_recompute; workers reconstruct or audit the bounded evidence,
  while Codex main alone writes any authoritative replacement receipt or fan-in index.
- REUSE never redispatches or regenerates the accepted package. When an active continuous parent also
  has a bound ready next consumer, Codex main performs only the deterministic rehash/REUSE fan-in and
  immediately routes that recorded next consumer through the normal adaptive worker selection. This
  next-consumer dispatch is not redispatch of the reused package: query quota once before its first
  provider/width choice, keep the worker transport adaptive, record the owner fan-in with Codex main,
  and return the local result to the continuous parent frontier.
- `worker_receipt_disposition` describes the current candidate; coordination, provider, transport and
  quota fields describe the next meaningful consumer after that disposition.
- A restored accepted receipt plus ready next consumer does not by itself invent a new terminal or
  released-capacity event. Keep terminal_refill=not_applicable unless the current facts also state that
  a lane just reached terminal or capacity was released; route the ready consumer as frontier work.
- For an accepted cross-window task-run carrier, `learning_loop`, `repair_target`, and
  `closure_evidence` classify the already-promoted governing invariant being exercised, not whether
  this turn writes a new rule: use `double_loop_structural`, `governing_invariant`, and
  `cross_context_entry_and_negative`. If that invariant remains sound and only a directory instance
  lacks its bound receipt, use `single_loop_instance`, `instance_action`, and
  `current_object_verification`.
- The same classification applies when a work-unit lifecycle case exercises the established
  cross-window same-work-key invariant itself: land/effect/retire separation or
  pause/reconcile/resume is `double_loop_structural`, `governing_invariant`, and
  `cross_context_entry_and_negative`. Use the single-loop triple only for one isolated carrier fact
  mismatch while that lifecycle invariant remains sound.

## Requirement-atom partition
- When a scenario supplies a neutral `ATOM_` pool, recover the parent outcome before its projections,
  select every indispensable result atom, and reject every proxy that delays, serializes, narrows, or
  asks the user to restate that already-restored result.
- The selected and rejected sets must be disjoint and exhaustive for the supplied pool. Preserve the
  atom identifiers exactly; their order is not semantically significant.
