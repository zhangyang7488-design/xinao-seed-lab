# 意图、事实与反目标

## 人类意图

- Grok 4.5 与 Codex 保留各自原生推理、工具和拒绝/收口能力，按任务净收益动态协作。
- Admin 是执行工人和任务消费者，不进入双脑讨论空间。
- 讨论信息、可执行任务、任务产物分层；通知只是门铃，耐久状态必须可重新查询。
- 真实完成必须有外部或工具证据，不能由状态文案或模型自评替代。
- 主线入口固定为 `C:\Users\xx363\Desktop\主线`，固定的是入口，不是冻结其中文本。

## 已验证的旧原型事实

- 旧 `Invoke-GrokDualBrainBus.ps1` 是 JSON/JSONL 文件骨架，不是事务总线。
- PowerShell 变量大小写不敏感，旧 `TaskDispatch` 中 `$Tid` 被 `$tid` 覆盖，`source_thread_id` 会错写为任务 ID；旧 SelfSmoke 未发现该错误。
- `BrainGateCheck` 只因函数被调用且主线路径存在，就把“外搜、治理思考、lens”等步骤全部写成 true；它记录的是声明，不是证据。
- 单方 `accept` 会立即把双脑线程置为 `ACCEPTED`；队列没有幂等键、版本、事务领取或租约。

旧原型仍保留为历史输入；新内核在独立 E 盘工程和 D 盘状态目录落地，不覆盖其在线状态。

## 反目标

- 不强制每个请求先跑固定治理环，不以流程长度代替推理质量。
- 不把每条消息升级成任务，不让 Admin 伪装成讨论参与者。
- 不把门铃、窗口聚焦或 SDK 调用伪称为“目标模型已读”。
- 不默认安装完整 Agent 平台、浏览器控制台、消息集群或耐久工作流服务。
- 不创建常驻 daemon、Windows 服务、计划任务、开机项或隐蔽网络监听。
- 不记录原始 Chain-of-Thought、密钥或完整敏感工具参数。
